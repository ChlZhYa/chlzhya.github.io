## 为什么又是向量检索

向量检索（vector search）早已不是新鲜事，但在生产环境里把它跑得"又准又快又稳"，依然充满了容易被低估的工程取舍。这篇文章记录我在一次 RAG 系统上线过程中的几关键决策。

## HNSW 的两个参数决定一切

HNSW（分层可导航小世界图）几乎是当下向量索引的事实标准，但它有两个参数会显著影响行为：

- **`M`**：每个节点的连接数。越大，召回越高、内存越大、构建越慢。
- **`efConstruction` / `efSearch`**：构建与查询时的搜索宽度。

一个常见误区是把 `M` 调到很大以为能"更准"。实际上，当向量本身质量不足时，更大的 `M` 只会让你在错误的近邻空间里更"自信"地搜索。

> 先把 embedding 质量做对，再调 HNSW 参数。

## 召回率必须离线评测

线上点击率不等于召回正确。在投入生产前，应该建一个**离线评测集**：

1. 人工标注一批 query 的"正确答案"
2. 对比不同参数下的 top-k 命中率
3. 固化一套 golden set，每次索引变更回归

没有这套评测，所有"调参"都只是玄学。

## 混合检索是性价比最高的一步

纯向量检索在"专有名词 + 通用词"混合的 query 上常常输给 BM25。把两者融合（hybrid search）通常能带来最明显的提升：

- BM25 负责**精确词项命中**
- 向量负责**语义相似**
- 用 RRF（Reciprocal Rank Fusion）或加权分数融合

在我遇到的场景里，仅加一层 BM25 混合，召回率就提升了约 18%。

## 小结

向量检索的工程化，难点不在算法本身，而在于：评测体系的建立、参数与 embedding 质量的匹配、以及混合策略的引入。把这些做扎实，比追逐最新的 ANN 论文更有回报。
